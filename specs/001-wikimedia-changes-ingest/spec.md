# Feature Specification: Wikimedia 변경사항 Bronze 적재 파이프라인

**Feature Branch**: `001-wikimedia-changes-ingest`

**Created**: 2026-05-21

**Status**: Draft

**Input**: User description: "wiki media changes api를 수집해서 volume에 저장하고 싶어. 5분마다 실행하도록. 형식은 가장 효율적인 상태로. 일단 bronze로 layer에 volume으로, 그 전에 configuration 폴더에 catalogs.yml으로 catalog 정의(이름은 wikimedia-dataplay). 스키마는 raws? 이건 모르겠다. 너 추천하는 방향으로."

## Clarifications

### Session 2026-05-22

- Q: 위키미디어 변경 데이터의 실제 소스로 어떤 공개 API 를 사용할 것인가? → A: MediaWiki Action API `list=recentchanges` (`/w/api.php`, REST 폴링, `rcstart/rcend` 윈도우, 멱등 키 `rcid`)
- Q: 적재 대상 wiki 프로젝트 집합은 어디까지인가? → A: 영문 위키피디아(`en.wikipedia.org`) 단일 wiki 만 (MVP)
- Q: 동일 윈도우 재실행 시 볼륨 파일의 멱등성 정책은? → A: 파일 덮어쓰기(overwrite) — 윈도우 경로의 파일을 무조건 덮어쓰고 상태 저장 없음. API 결정성에 의존.
- Q: 5분 윈도우의 정렬 기준은? → A: 벽시계 정렬 — `[HH:00, HH:05), [HH:05, HH:10), ...` 슬롯. 잡은 매 정각/5/10/... 분에 트리거되어 직전 슬롯을 처리. 윈도우 키 = 시작 시각.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Bronze 레이어에 위키미디어 변경사항 정기 적재 (Priority: P1)

데이터 분석가/엔지니어가 위키미디어 위키들에서 발생하는 모든 변경사항(편집·생성·삭제·로그 이벤트 등)을 분석에 활용하기 위해, 위키미디어 공식 변경 피드를 5분 주기로 수집해 dataplay 레이크하우스의 bronze 레이어 볼륨에 **원본 그대로** 적재한다.

**Why this priority**: 본 데이터셋은 후속 silver/gold 가공, 분석, ML 학습의 공통 입력이다. 이 스토리 없이는 어떤 후속 작업도 불가능하므로 MVP.

**Independent Test**: 잡을 1회 실행하면 직전 5분 윈도우의 변경 이벤트가 지정된 볼륨 경로에 적재되고, 적재 파일을 다시 읽었을 때 원본 페이로드가 그대로 보존되어 있어야 한다.

**Acceptance Scenarios**:

1. **Given** 위키미디어 변경 피드가 정상 동작 중이고, **When** 잡이 정기 스케줄(5분)에 따라 실행되면, **Then** 직전 5분 윈도우의 모든 변경 이벤트가 누락 없이 볼륨에 저장된다.
2. **Given** 동일 윈도우가 두 번 처리되는 상황(재시도/수동 재실행), **When** 잡이 다시 실행되면, **Then** 동일 이벤트가 중복 적재되지 않거나(권장) 최소한 중복 식별이 가능한 형태로 저장된다.
3. **Given** 외부 피드가 일시적으로 응답 불가 상태일 때, **When** 잡이 실행되면, **Then** 잡은 실패로 기록되고 다음 정기 실행에서 가장 최근 누락 윈도우를 보정 시도한다.
4. **Given** 적재된 파일을 데이터 소비자(분석가)가 읽을 때, **When** 임의 윈도우의 파일을 로드하면, **Then** 윈도우 시작·종료 시각으로 경로/파일명을 식별할 수 있다.

---

### User Story 2 - 카탈로그·스키마·볼륨을 코드로 정의 (Priority: P2)

본 잡이 적재 대상으로 삼는 카탈로그(`wikimedia-dataplay`), 스키마(권장: `bronze`), 적재용 볼륨이 환경 구성 파일에 의해 코드 형상관리되어 있어야, 환경(local/lab/...) 간 일관성과 거버넌스(권한, 위치, 명명)가 유지된다.

**Why this priority**: US1 의 선행 조건이지만 단독 검증 가능. 데이터 자산 정의가 코드로 박혀 있어야 어떤 환경에서도 동일한 구조로 재현 가능하고, 워크스페이스 UI 에서의 수동 변경(drift)을 막을 수 있다.

**Independent Test**: 새 환경(예: 새 workspace target)에 번들을 배포했을 때, `configuration/catalogs.yml` 만으로 카탈로그·스키마·볼륨이 자동 생성되고 동일 이름/구조로 워크스페이스에 나타난다.

**Acceptance Scenarios**:

1. **Given** 새 환경에서 자산이 아직 존재하지 않을 때, **When** 번들 deploy 가 수행되면, **Then** `wikimedia-dataplay` 카탈로그·`bronze` 스키마·적재용 볼륨이 단일 deploy 로 생성된다.
2. **Given** `configuration/catalogs.yml` 만 변경한 PR 에서, **When** `bundle validate` 가 실행되면, **Then** 변경 내역이 plan 단계에서 명확히 표시되고 검증을 통과한다.
3. **Given** 워크스페이스 UI 에서 같은 이름의 자산이 수동 생성/변경된 경우, **When** 번들 deploy 가 다시 실행되면, **Then** 코드 정의가 단일 소스로서 우선시되고 drift 가 감지·기록된다.

### User Story 3 - 적재 결과의 운영 관측성 (Priority: P3)

운영자/데이터 엔지니어가 본 파이프라인의 건강 상태(성공/실패, 적재 레코드 수, 지연)를 즉시 확인할 수 있어 데이터 결손에 빠르게 대응한다.

**Why this priority**: 데이터 신뢰성을 위한 부가가치. MVP 이후 단계에서 추가해도 본 기능의 핵심 가치(적재) 는 동작한다.

**Independent Test**: 임의 잡 실행 후 잡 로그에 처리한 윈도우 시각과 적재 레코드 수가 노출되고, 실패 시 기존 운영 알림 채널(Discord) 로 메시지가 발송된다.

**Acceptance Scenarios**:

1. **Given** 잡이 정상 종료된 직후, **When** 운영자가 잡 실행 로그를 조회하면, **Then** 처리한 윈도우(시작·종료 시각)와 적재 레코드 수가 구조화된 로그로 노출된다.
2. **Given** 잡이 실패한 직후, **When** 운영자가 Discord 알림을 확인하면, **Then** 잡 식별자·실패 윈도우·에러 요약이 포함되어 있다.

---

### Edge Cases

- **윈도우 경계의 이벤트**: 변경 이벤트 타임스탬프가 윈도우 경계에 정확히 걸친 경우의 포함/배제 규칙은 일관되어야 한다(예: `[start, end)` 좌폐우개 규약).
- **백필 한도**: 잡이 일정 시간(예: 1시간) 이상 중단된 후 재개될 때 자동 보정의 최대 윈도우 수, 그 이상은 어떻게 처리할지 명확해야 한다.
- **이벤트 폭증**: 평소 대비 10배 이상의 이벤트가 1 윈도우에 몰려 적재 파일이 비대해질 때 단일 파일/다중 파일 정책.
- **외부 피드의 페이로드 스키마 변경**: 위키미디어 측 페이로드에 새 필드/타입 변경이 있을 때 적재 무결성이 깨지지 않아야 한다(원본 보존 형식의 이점).
- **부분 수신 실패**: 외부 호출이 중간에 끊겨 윈도우 일부만 받은 경우, 적재 파일을 partial 로 표시하거나 폐기하고 다음 실행에서 재시도.
- **시계 차이/피드 지연**: 외부 피드의 시각과 잡 실행 시점의 시각 사이 lag 가 큰 경우, 윈도우 기준을 피드 timestamp 로 둘지 잡 실행 시각으로 둘지 명확해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 **MediaWiki Action API** `list=recentchanges` 엔드포인트를 통해 **영문 위키피디아**(`https://en.wikipedia.org/w/api.php`) 의 변경 이벤트(편집·생성·삭제·로그)를 시각 윈도우(`rcstart`/`rcend`) 단위로 수집해야 한다. 멱등 식별자는 `rcid` 를 사용한다. 대상 wiki 집합은 코드(설정 파일) 한 곳에서 관리되어 향후 확장 가능해야 한다.
- **FR-002**: 시스템은 매 정각 및 매 5/10/... 분(`*/5 * * * *`) 에 자동 트리거되어 **직전 벽시계 슬롯** (`[HH:MM, HH:(MM+5))`) 을 윈도우로 적재해야 한다. 윈도우 키는 시작 시각이며 경계는 좌폐우개 `[start, end)`.
- **FR-003**: 시스템은 수집한 이벤트를 **가공 없이 원본 페이로드를 보존**하여 Databricks 볼륨에 저장해야 한다.
- **FR-004**: 시스템은 디스크/네트워크 효율을 위해 **압축된 라인 기반 JSON(NDJSON + gzip)** 형식으로 저장해야 한다. 형식 변경은 plan 단계에서 명시적 정당화가 있을 때만 허용한다.
- **FR-005**: 시스템은 적재 대상 카탈로그(`wikimedia-dataplay`)·스키마(`bronze`)·볼륨을 `configuration/catalogs.yml` 에서 코드로 정의해야 한다.
- **FR-006**: 시스템은 동일 윈도우 재실행 시 **윈도우 경로의 파일을 덮어쓰기(overwrite)** 함으로써 멱등성을 보장해야 한다. 별도 외부 상태 저장은 사용하지 않는다(API 결정성에 의존).
- **FR-007**: 시스템은 외부 피드 일시 장애 시 잡을 실패로 처리하고, 다음 정기 실행에서 가장 최근 누락 윈도우의 보정을 시도할 수 있어야 한다.
- **FR-008**: 잡 실패 시 운영 채널(Discord) 로 알림이 발송되어야 한다(기존 CI/CD 알림 채널 재사용).
- **FR-009**: 카탈로그·스키마·볼륨 정의는 환경(target) 별 파라미터화가 가능해 단일 소스(`configuration/catalogs.yml`) 에서 환경별 차이가 표현되어야 한다.
- **FR-010**: 적재 파일은 윈도우 시작 시각 기반 경로/명명 규약(예: `…/year=YYYY/month=MM/day=DD/hour=HH/minute=MM/<file>`) 으로 저장되어 벽시계 5분 슬롯을 경로로 유일하게 식별할 수 있어야 한다.
- **FR-011**: 잡 실행 로그는 처리한 윈도우 시각과 적재 레코드 수를 포함해야 한다(US3, P3).

### Key Entities *(데이터 관련)*

- **변경 이벤트(Change Event)**: 위키미디어 위키에서 발생한 단일 액션의 원본 페이로드. wiki 식별자·시각·행위자·페이지·변경 유형 등을 포함.
- **적재 윈도우(Ingest Window)**: 5분 길이의 시간 구간. 잡 1회 실행 = 1 윈도우 처리.
- **적재 파일(Ingest File)**: 1 윈도우의 모든 이벤트를 담은 결과 산출물. 볼륨 내 시간 기반 경로에 저장.
- **카탈로그(`wikimedia-dataplay`)**: Unity Catalog 의 최상위 자산. 본 도메인의 데이터 자산을 묶는 단위.
- **스키마(`bronze`)**: 카탈로그 하위 데이터 레이어 단위. 원본 보존 영역.
- **볼륨(Volume)**: 스키마 하위에서 파일을 보관하는 자산. 본 잡이 이벤트 파일을 적재하는 대상.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 5분 주기 스케줄로 1주(168시간) 연속 운영 시 적재 윈도우 누락률이 1% 이하다.
- **SC-002**: 1회 실행(=1 윈도우 처리) 이 정상 부하에서 3분 이내에 종료된다(다음 실행과 겹침 방지).
- **SC-003**: 적재 데이터의 원본 페이로드가 100% 보존된다(이벤트 수와 페이로드 본문이 원본 피드와 일치).
- **SC-004**: 새 환경에 번들을 배포한 직후 30분 이내에 카탈로그·스키마·볼륨이 준비되고 첫 윈도우가 적재된다.
- **SC-005**: 잡 실패 발생 시점부터 운영자가 알림을 받기까지 평균 5분 이내.
- **SC-006**: 1일 단위 적재 비용(저장+컴퓨트) 이 운영팀이 합의한 허용 구간 이내로 유지된다.
- **SC-007**: 데이터 소비자(분석가) 가 임의 5분 윈도우의 파일을 1분 이내에 식별·열람할 수 있다(시간 기반 경로 규약 덕분).

## Assumptions

- **데이터 소스**: **MediaWiki Action API** `list=recentchanges` (공개 REST, 인증 불필요) 를 사용한다. 멱등 키는 응답에 포함된 `rcid` (각 wiki 내에서 monotonic 정수).
- **스코프**: MVP 는 **영문 위키피디아 단일 wiki** (`en.wikipedia.org`). 대상 wiki 목록은 설정 파일(예: `configuration/wikis.yml` 또는 `catalogs.yml` 의 파라미터) 로 코드 관리해 향후 확장한다. 다른 wiki/언어판 추가는 본 기능 범위 외.
- **저장 형식 권장**: **NDJSON + gzip** (라인당 1 이벤트, 1 윈도우당 1 파일 또는 소수 파일). 이유: (a) 스키마 진화에 강건, (b) 원본 보존, (c) 압축률 우수, (d) Databricks 에서 Auto Loader/Spark 로 직접 읽기 용이.
- **스키마 이름 권장**: **`bronze`** (medallion 아키텍처 규약). 사용자가 언급한 `raws` 대신 단수형 `bronze` 가 후속 `silver`·`gold` 추가 시 명명이 자연스러움.
- **카탈로그 이름**: 사용자가 지정한 `wikimedia-dataplay` 그대로(Unity Catalog 식별자에 하이픈 허용, 참조 시 백틱).
- **볼륨 이름 권장**: `recentchanges_raw` (또는 동등 명명). MANAGED 볼륨.
- **운영 알림 채널**: 기존 `DATABRICKS_BUNDLES_DISCORD_WEBHOOK_URL` 채널 재사용.
- **운영 환경**: `targets.lab` 워크스페이스에서 우선 운영. prod 분리는 향후 결정.
- **PII/거버넌스**: 위키미디어 공개 데이터로서 마스킹/익명화 불필요.
- **재시도 정책**: 단일 실패는 다음 정기 실행에서 가장 가까운 누락 윈도우 1개 보정. 장기간(예: 6시간 이상) 중단 시 백필 정책은 별도 운영자 결정.
- **외부 의존성**: 위키미디어 측 가용성·rate limit 준수. SLA 가 보장되는 외부 자원이 아님을 인지.
- **카탈로그 사전조건**: 첫 deploy 시 워크스페이스 SP 에 `CREATE CATALOG` 권한이 부여되어 있다고 가정(아니라면 사전 발급 필요).
